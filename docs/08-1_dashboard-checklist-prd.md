# 08-1 대시보드 체크리스트 PRD

## 1. 개요
- 대시보드 우측 상단에 고정된 초간단 체크리스트 위젯을 추가해 사용자가 당일(또는 단기) 할 일을 비워낼 수 있도록 한다.
- 기존 스티커 메모 위젯의 최대 슬롯을 4개에서 3개로 줄이고, 가장 우측 영역을 체크리스트 위젯이 항상 차지하도록 레이아웃을 재조정한다.
- 본 문서는 기능 정의 및 UX 요구사항까지만 다루며, 구현 계획이나 세부 기술 설계는 범위에 포함하지 않는다.

## 2. 목표 및 성공 지표
1. 매일 대시보드 접근 시 즉시 확인 가능한 단기 Todo 영역 확보.
2. 사용자 1명이 동시에 관리할 수 있는 체크 항목 수를 5~7개로 유지해 과부하를 방지.
3. 체크/해제, 항목 추가/삭제가 각각 1번의 클릭 혹은 키 입력으로 완료될 만큼 단순한 UX 제공.

### 2.1 성공 지표 (향후 계측 가정)
- 체크리스트에 최소 1개 이상의 항목을 보유한 활성 사용자 비율 ≥ 60%.
- 하루 동안 완료된 체크 수 / 추가된 체크 수 ≥ 0.7 유지.
- 체크리스트 인터랙션(추가/체크/삭제) 중 오류 재시도율 < 1%.

## 3. 주요 사용자 시나리오
1. **빠른 할 일 작성**: 아침에 대시보드 진입 → 우측 체크리스트에서 빈 입력 칸 클릭 → "OKR 리포트 작성" 입력 후 Enter → 자동 저장.
2. **완료 처리**: 업무를 끝낸 후 체크박스를 클릭하면 항목이 지워지지 않고 줄이 그어지며 리스트 하단으로 이동하거나 페이드 처리되어 성취감을 제공.
3. **정리**: 필요없는 항목의 휴지통 아이콘을 눌러 삭제 → 삭제는 즉시 반영되며 되돌리기는 미지원.

## 4. 기능 요구사항

### 4.1 레이아웃 및 위치
- 대시보드 우측 상단 고정 폭(예: 280~320px) 카드.
- 스티커 메모 3장과 한 행을 공유하며, 반응형일 때도 순서가 보장되어 항상 가장 우측에 위치.

### 4.2 체크 항목 구성
- 각 항목: 체크박스 + 단일 행 텍스트 입력 + 삭제(ActionIcon).
- 텍스트 길이 60자 제한, 초과 시 말줄임 처리와 Tooltip 제공.
- 미완료 항목은 입력 순 정렬, 완료 항목은 카드 하단 영역에 축소 표시(예: “완료 2개 보기” 토글).

### 4.3 인터랙션
- 입력 엔터 시 새 항목 추가, 빈 값 제출 시 방지.
- 체크 상태 토글 시 0.3초 이내 피드백(상태 색상/줄긋기).
- 삭제 시 별도 모달 없이 즉시 제거, 단 1초 동안 Undo 스낵바 제공 검토.

### 4.4 저장 정책
- 모든 조작은 자동 저장(onChange)으로 서버 반영.
- 네트워크 실패 시 토스트 + 재시도 버튼, 실패 항목은 로컬 상태 유지.
- 5~7개 제한(정확한 수치는 UX 검증 후 확정). 제한 도달 시 입력창 비활성 및 안내 텍스트 표시.

### 4.5 접근성/언어
- 키보드 전용 조작: Tab 이동, Space로 체크 토글, Delete 키로 삭제 지원.
- 한글 입력 중 엔터 제출 시 조합 종료 후 처리 (IME 대응).

## 5. 기술 및 데이터 고려사항
- **프론트엔드**
  - 현존 대시보드 레이아웃(Grid) 내 새로운 `ChecklistWidget` 컴포넌트 추가.
  - TanStack Query 기반 API 연동, Optimistic Update 적용 검토.
  - Mantine `Checkbox`, `TextInput`, `Stack` 조합으로 단순 구현.
- **백엔드**
  - `ChecklistItem` 모델(예: id, content, isCompleted, order, userId, createdAt, updatedAt).
  - REST 엔드포인트: `GET/POST /api/checklist`, `PATCH/DELETE /api/checklist/:id`.
  - 완료 항목 자동 정렬 및 order 필드 관리 필요.
- **데이터 마이그레이션**
  - 기존 사용자에게는 빈 리스트 자동 생성(서버 응답에서 배열 제공).

## 6. 의존성 및 고려해야 할 변경
1. **스티커 메모 슬롯 감소**: 기존 4개 저장하던 사용자에 대한 UX 안내 필요(예: 4번째 메모는 자동 숨김? 또는 선택적으로 삭제 유도). 임시 스펙: 최신 순 3개만 노출, 나머지는 읽기 전용 모달에서 확인 가능하도록 추후 결정.
2. **반응형 레이아웃**: 모바일/태블릿에서 체크리스트가 스티커 메모 아래로 내려가더라도 위치는 고정 섹션으로 표시.
3. **상태 초기화**: 매일 0시 자동 리셋 여부? → 이번 범위에서는 수동 관리(사용자가 직접 삭제/체크)로 결정.

## 7. 오픈 이슈
1. 완료 항목 자동 숨김 vs. 즉시 삭제 중 어떤 경험이 더 적합한지 사용자 피드백 필요.
2. Sticky Notes -> Checklist 전환으로 인한 시각적 밀도 증가를 어떻게 완화할지(간격, 색상 톤 조정) UI 시안 요구.
3. 최대 항목 수(5 vs 7)와 완료 항목 표시 방식 A/B 필요.

## 8. 비범위(Out of Scope)
- 다중 보드/프로젝트별 체크리스트.
- 마감일, 우선순위, 태그 등 복잡한 Task 속성.
- 알림, 리마인더, 캘린더 동기화.
- 팀 공유/협업 체크리스트.

## 9. 차후 단계 제안
1. 와이어프레임 확정 → 스티커 메모/체크리스트 병치 시안 리뷰.
2. UX 검증(사용자 3~5명 인터뷰)으로 항목 수·완료 처리 방식 결정.
3. API/데이터 모델 설계, 구현 계획서는 `08-2` 문서로 분리 작성.
